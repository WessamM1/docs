## [Overview](#overview)

A policy governs resource access within an [org](/api-reference/org) to a set of principals. It enables fine-grained authorization rules to define the
minimum amount of permissions required when accessing resources of the platform.

A policy consists of:

- A resource, and
- One or more bindings

### [Resource](#resource)

A resource is a Control Plane object (e.g., [secret](/api-reference/secret), [workload](workload), [GVC](/api-reference/gvc), etc.).

A policy can be configured to target all or specific resources within your [org](/api-reference/org).

For example, a policy can target all the [GVCs](/api-reference/gvc) within your [org](/api-reference/org), or specifically GVC A and GVC B.

Specific resources can be assigned directly or dynamically (using a [query](/api-reference/misc#tags)).

### [Bindings](#binding)

A binding is a mapping between:

- A set of permissions (e.g., create, delete, etc.), and
- Principal membership

The [set of permissions](/api-reference/policy#permissions) that can be assigned to a policy are unique to each resource.

Principals can be [users](/api-reference/user), [groups](/api-reference/group), [service accounts](/api-reference/serviceaccount), and [identities](/api-reference/identity).

## [Examples](#examples)

- Allow a [user](/api-reference/user) or [group](/api-reference/group) access to the console
- Allow a [service account](/api-reference/serviceaccount) to execute CLI commands
- Allow an [identity](/api-reference/identity) access to reveal a secret. The [identity](/api-reference/identity) can then be associated with a [workload](workload). The workload's containers will have permissions
  to access the [secret](/api-reference/secret) and use it as an environment variable.
